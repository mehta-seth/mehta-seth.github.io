# aditmehta-site

Personal website of Adit Mehta. Built with Astro, self-hosted fonts,
dual-theme with a View Transitions clip-path reveal animation on theme
toggle.

This delivery is **Phase 1 only** (scaffold + typography + theming
foundation). See `adit-mehta-website-v1-plan.md` §5 for the full phase
breakdown. Phase 2 onwards is intentionally not started.

---

## Run it

```bash
npm install
npm run dev        # http://localhost:4321 — hot reload, dev server
npm run build      # static output in /dist
npm run preview    # serve /dist locally at http://localhost:4321
```

Requires Node 20+ (Node 22 tested).

## What Phase 1 delivers

**Infrastructure**

- Astro 5.18 static-site scaffold, TypeScript strict mode.
- Integrations: `@astrojs/tailwind`, `@astrojs/mdx`, `@astrojs/sitemap`.
- Markdown plumbing for Phase 3 already wired: `remark-smartypants`,
  `rehype-slug`, `rehype-autolink-headings`.

**Fonts (self-hosted, no third-party runtime fetch)**

Six WOFF2 files in `/public/fonts/`, pre-subsetted by language:

| File                              | Role                          | Size  |
| --------------------------------- | ----------------------------- | ----- |
| `newsreader-400.woff2`            | Body serif, regular           | 14 KB |
| `newsreader-400-italic.woff2`     | Body serif, italic            | 15 KB |
| `newsreader-600.woff2`            | Body serif, semibold          | 15 KB |
| `kalam-devanagari-400.woff2`      | Devanagari wordmark, regular  | 109 KB |
| `kalam-devanagari-700.woff2`      | Devanagari wordmark, bold     | 108 KB |
| `jetbrains-mono-400.woff2`        | Inline code                   | 21 KB |

Kalam uses a `unicode-range` declaration (in `src/styles/global.css`) so
it's only invoked for Devanagari codepoints — never for Latin body text.

The two most-important fonts (Newsreader regular, Kalam Devanagari
regular) are `<link rel="preload">`-ed in `<head>` to eliminate
flash-of-unstyled-text on first paint.

**Theming system**

- Design tokens live in `src/styles/global.css` as CSS custom properties,
  defined twice: once on `:root` (light) and once on `:root.dark` (dark).
  Tailwind's color utilities reference these vars via
  `tailwind.config.mjs`, so `bg-bg`, `text-ink`, etc. follow the active
  theme automatically.
- Default theme on first visit: **dark**, overriding the OS
  `prefers-color-scheme`. User choice is persisted in `localStorage` under
  the key `theme`.
- The `<html class="dark">` class is set by a **synchronous inline
  script at the top of `<head>`** (`BaseLayout.astro`), before the
  browser paints. No flash of wrong theme on refresh.

**Theme toggle**

- Fixed-position button, top-right of every page.
- Hand-drawn sun + moon SVGs (not a Lucide import) with stroke weights
  tuned to the site's text.
- On click: View Transitions API + a `clip-path` circle that expands
  from the click point to the far corner of the viewport, revealing the
  new theme. 500ms, `cubic-bezier(0.4, 0, 0.2, 1)` easing.
- Graceful fallbacks:
  - No View Transitions support (current Firefox) → instant class swap
    + the `<body>`'s 200ms background-color transition as a soft fade.
  - `prefers-reduced-motion: reduce` → same instant swap.
  - Keyboard-triggered click → origin defaults to the button's center
    rather than `(0, 0)`.

**Grain overlay (§3.6 of the plan)**

- 200×200 px seamless-tiling grain PNG in `/public/textures/grain.png`,
  generated by `scripts/generate-grain.py` with a deterministic seed (so
  the grain is identical on every regeneration).
- Applied as a fixed-position `body::before` pseudo-element at variable
  opacity: 4% in dark mode, 1.5% in light mode. `mix-blend-mode: overlay`
  for tone integration. `pointer-events: none` so it never intercepts
  clicks.

**Placeholder home page**

- `src/pages/index.astro` shows only the wordmark and theme toggle plus
  a muted "Scaffold ready" note. This is the Phase 1 deliverable per the
  plan's §5; the real home page content (bio, essays list, cities list,
  email) is Phase 2 and will replace this file's body.

## Known limitations flagged for later phases

1. **Kalam WOFF2 size (~108 KB per weight).** Exceeds the §6 per-font
   budget of 40 KB. Inherent to Devanagari scripts — the conjunct and
   ligature tables are large. If Phase 5's Lighthouse pass shows this as
   a real problem, options are: (a) drop weight 700, (b) subset Kalam
   further to only the glyphs actually used in "अदित मेहता" (aggressive
   but feasible if the wordmark is the only Devanagari on the site).
2. **Build warning about auto-generated `essays` collection.** Astro 5
   expects `src/content.config.ts` instead of `src/content/config.ts`.
   Cosmetic; fixed in Phase 3 when the collection is populated.
3. **Favicon is a 1×1 transparent inline SVG.** Prevents 404s but isn't
   a real favicon. Phase 5 ships one.
4. **Empty `src/assets/cities/` and `src/content/essays/` directories.**
   Present because the plan's §4.4 calls for them. Populated in
   Phases 3 and 4.

## Font acquisition note

The plan's §3.4 specifies "download from Google Fonts, subset with
`glyphhanger`." This delivery uses the `@fontsource/*` npm packages as
the source for the WOFF2s — they ship identical Google Fonts files,
pre-subsetted by language. The files were copied into `/public/fonts/`
and the npm packages were uninstalled, so the end state is exactly what
the spec asks for: six self-hosted WOFF2s, no third-party runtime
request, no `glyphhanger` dependency in the repo.

If you prefer the manual workflow for reproducibility, replacing the
files with the output of `glyphhanger` is a drop-in swap — the
`@font-face` declarations in `src/styles/global.css` reference files by
path only.

## Git: push to your GitHub repo

From inside this project directory:

```bash
git init
git add .
git commit -m "Phase 1: scaffold, typography, theming foundation"
git branch -M main
git remote add origin https://github.com/mehta-seth/mehta-seth.github.io.git
git push -u origin main
```

If the repo already has a commit (e.g. an auto-generated README from
GitHub's "Initialize this repository" toggle), either:

- **Easiest**: delete the repo and recreate it without initializing any
  files, then run the commands above.
- **Alternative**: `git pull origin main --allow-unrelated-histories`
  before the push, then resolve any conflicts in the README.

The site won't be live until Phase 6 (which adds the GitHub Actions
deploy workflow and configures Pages in repo settings).
